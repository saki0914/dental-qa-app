# Dental QA App 作業ルール

このファイルはリポジトリ全体に適用する。

## 修正・実装の標準フロー

- ユーザーから修正または実装を依頼されたら、調査、実装、検証、`main`への反映までを一連の作業として扱う。
- 作業開始時に現在のブランチと未コミット差分を確認し、ユーザーの既存変更を上書きしたり、今回と無関係な差分をコミットしたりしない。
- 実行環境は`.devcontainer/`の定義を優先する。ホスト側にJavaやブラウザ依存がない場合も、そこで検証を止めず、現在のDev Containerを起動または再ビルドして続行する。
- 実装完了後はDev Container内で`npm run test:all`を実行する。これには構文検査、単体テスト、Firestore/Storageルール、未ログインE2E、Firebase Emulatorを使う認証付きE2Eが含まれる。
- テストが失敗した場合は原因を修正して再実行する。全テスト成功前に完了扱い、コミット、pushをしない。
- UI変更では必要に応じてDev Container内で`npm run dev`を起動し、ユーザーへ`http://localhost:3000`を提示する。認証付きのローカル確認にはFirebase Emulatorを起動し、`http://localhost:3000/?firebaseEmulator=1`を提示する。外部公開URLにはしない。
- 本番Firebaseをテストに使用しない。Emulatorは`demo-dental-qa`を使用する。

## Git反映

- 修正・実装依頼では、ユーザーが「コミットしない」「pushしない」「PRにする」など別の指定をしていない限り、検証済みの今回差分だけを`main`へコミットし、`origin/main`へpushする。
- `main`以外で作業している場合は、差分と競合を確認してから`main`へ安全に統合する。
- force push、履歴改変、無関係な変更の取り消しは行わない。
- Firebase Hostingや本番ルールのデプロイは、明示的に依頼された場合だけ行う。`main`へのpushと本番デプロイは別の操作として扱う。

## 完了報告

- 実際に実行したテストと件数、失敗・スキップの有無を報告する。テストコードが存在するだけで「実行済み」と表現しない。
- コミットハッシュ、`main`と`origin/main`の同期状態、残っている未コミット差分を報告する。
- 実行できなかった検証がある場合は、理由と未検証範囲を明記する。

## 調査のみの依頼

- ユーザーが説明、診断、レビュー、状況確認だけを依頼した場合は、明示的な修正依頼がない限り、ファイル変更、コミット、pushを行わない。
